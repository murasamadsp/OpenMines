## Методология сервера: server-authoritative (Fortnite/Quake-style)

Применяется **ко всему** игровому взаимодействию: movement, dig, build, inventory, packs, экономика, скиллы, клан-действия. Любой новый handler должен ей следовать.

### Базовая модель

- **Сервер — единственный источник правды.** Всё игровое состояние живёт в `GameState` (`active_players`, `world`, `db`, `packs`). Клиент никогда не является авторитетом.
- **Клиент шлёт intent, не факт.** `TY Xmov/Xdig/Xbld/...` — это запрос намерения с его текущим видением (dest, dir, target). Сервер сам решает, исполнять или отклонить.
- **Client-side prediction.** Клиент мгновенно применяет действие локально (двинул бота, снял HP с блока), чтобы UI был отзывчив.
- **Reconciliation.** Если сервер отклоняет — присылает корректирующий снапшот (`@T`, повторный HB, `@L` HP, `@B` корзина). Клиент обязан снапнуть своё состояние по этому пакету.
- **Broadcast соседям с exclude-self.** Успешная операция рассылается окружению через `broadcast_to_nearby(..., exclude_id = Some(pid))`. Отправителю эхо собственных изменений не шлём — он уже предсказал.

### Единый pipeline handler'а

Каждый `TY X*` handler следует шагам:

1. **Resolve player** (`active_players.get_mut(&pid)`) + проверка живости и окна (`current_window is None`).
2. **Rate-limit** по типу действия (`last_move_ts`, `last_dig_ts` и т.д.). Нарушение → **тихий drop**, без корректирующих пакетов. Это анти-флуд, а не откат.
3. **Валидация содержимого intent** (координаты в мире, смежность, целевая клетка, ресурсы, права, клан-доступ).
4. **Commit** изменения в `GameState` / `world` / `db`.
5. **Outbound**:
   - Отправителю — только дифф состояния, если он отличается от предсказания (новая HP, новая корзина, новый инвентарь). Нет дифф — нет пакета.
   - Соседям — HB/`O`/`FX` c `exclude_id = Some(pid)`.
6. **Rejection** (1, 3) → корректирующий снапшот отправителю (`@T`, `@L`, `@B`, ...) + `warn!` с причиной. **warn**, не debug — это сигнал рассинхрона/читов.

### Правила отклонения и корректировки

| Нарушение | Действие сервера |
| - | - |
| Неактивный игрок / окно открыто / мёртв | `@T old` + `warn!`, закрыть окно если надо |
| Невалидные координаты / не-смежная клетка | `@T old` + `warn!` |
| Клетка занята / нет ресурсов / нет прав | корректирующий снапшот + `warn!` |
| Rate-limit (слишком часто) | см. ниже — НЕ silent drop |
| Ошибка БД / внутренняя | `@T old` + `error!`, обязательно логировать |

### ⚠️ Rate-limit и client prediction — несовместимы при silent drop

**Учённая ошибка**: серверный rate-limit с тихим дропом пакетов ломает client prediction.
Сценарий: клиент шлёт 5 ходов подряд (автодвижение/быстрые клавиши).
Сервер принимает первый, следующие 4 тихо дропает по rate-limit. Клиент тем временем
предсказал 5 клеток вперёд. Пятый ход проходит rate-limit → валидация смежности падает
(`|Δ| > 1`) → `@T` откат клиента на старую позицию.

**Правильно**: rate-limit делает клиент (`AddTime(pause)` в `ClientController`).
Сервер доверяет клиенту по частоте, валидирует только **содержимое** (смежность,
коллизии, ресурсы, права). Если нужен анти-чит по скорости — делать его **на основе
дистанции во времени**, а не дропом пакетов, и всегда с корректирующим `@T` (не silent).

### Инварианты

- Любое поле `PlayerData` (x/y, health, crystals, skills, inventory) меняется **в одной точке кода** на операцию. Не дублировать запись в нескольких местах.
- Не слать отправителю и корректирующий, и broadcast-эхо одновременно — это даёт визуальный «откат».
- Корректирующие снапшоты (`@T`, `@L`, `@B`) **не используются** для периодической синхронизации. Только в ответ на reject или события (респ, админ-TP, вход в пак).
- Все инварианты состояния валидируются на входе handler'а, не разбрасываются по логике.

### Логирование (продакшн)

- `RUST_LOG=openmines_server=info` — базовый уровень.
- `warn!` — любой reject, любая несогласованность клиент↔сервер.
- `debug!` — принятые действия (много спама, только при расследовании).
- Каждый `warn!` rejection должен содержать `pid`, тип действия, причину, координаты/цель.

### Проверка нового handler'а

Перед коммитом нового `TY`/чат-команды/pack-interaction:

- [ ] Есть rate-limit?
- [ ] Все rejection → `warn!` + корректирующий снапшот?
- [ ] Broadcast использует `exclude_id = Some(pid)`?
- [ ] Отправителю не шлётся эхо того, что он уже предсказал?
- [ ] Состояние пишется в одну точку?
- [ ] Проверка `current_window` и живости игрока?
