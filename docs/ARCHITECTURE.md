# Архитектура проекта

## Общая модель

Сервер строится по модульной модели:

- `config` — чтение `config.json` и логирование (`server/config.rs`, `server/logging.rs`).
- `world` — создание мира и загрузка ячеек (`server/world/`).
- `db` — инициализация и миграции SQLite (`server/db/mod.rs`).
- `game` — состояния игроков, крафт/навыки/чатовые сообщения (`server/game/`).
- `net` — входящие TCP-соединения, сессии и маршрутизация сообщений (`server/net/`).
- `protocol` — сериализация и десериализация пакетов (`server/protocol/`).

## Поток запуска

1. `main` грузит `config.json`; если файла нет — пишет дефолт в корень.
2. Инициализируется `tracing` через `config.logging`.
3. Параметр `--regen`/`--regen-world` или `M3R_REGEN_WORLD` удаляет старые `.mapb`.
4. Загружаются `cells.json` и создается `World`.
5. Инициализируется БД (`openmines.db`) и выполняются миграции.
6. Запускается `net` listener на `0.0.0.0:<port>`.
7. Для каждого клиента создаётся сессия, где идёт авторизация, инициализация игрока и обмен пакетами.

## Ключевые артефакты данных

- `config.json` — runtime-конфигурация сервера.
- `cells.json` — определения ячеек мира, загружаются при старте.
- `openmines.db` — SQLite с игроками, чатами, кланами и строительством.
- `*.mapb` — бинарные дампы мира (генерируются при `world::World::new`).

## Взаимодействие с клиентом

Серверный клиентский протокол реализован на уровне пакетов `server/protocol` и обрабатывается `server/net/session`.
Клиент-семантика изменяется в Unity-части, но любые изменения протокола требуют обновления и клиентских обработчиков, и этой документации.
