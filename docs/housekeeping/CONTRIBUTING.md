# Руководство для контрибьюторов

## Быстрый старт для нового участника

1. Сделайте fork репозитория и склонируйте его к себе.
2. Создайте ветку: `feat/...`, `fix/...` или `chore/...`.
3. Обновите релевантную документацию вместе с кодом.
4. Запустите проверки перед push.
5. Откройте Pull Request в `main`.

## Как запускать и проверять проект

### Настройка окружения

```bash
# базовая конфигурация
cp config.example.json config.json
```

### Основные команды разработки

```bash
cargo build --release
cargo test --all-targets --all-features
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
```

### Полный локальный quality-loop

```bash
bash scripts/ci-quality.sh
```

Команда выше выполняет форматирование, clippy, проверки зависимостей, тесты и сборку документации.

## Как оформить PR

- Указывайте **цель изменения**, проблему и ожидаемый эффект.
- Не меняйте Unity временные каталоги (`Library`, `Temp`, `obj`) без необходимости.
- Для изменений в протоколе сети отдельно опишите совместимость с клиентом.
- Если меняете API, обновите `docs/PROTOCOL.md`.
- Если меняете поведение запуска/конфига — обновите `README.md` и `docs/API.md`.

## Стиль коммитов

Рекомендуется использовать [Conventional Commits](https://www.conventionalcommits.org/):

- `feat: ...`
- `fix: ...`
- `docs: ...`
- `refactor: ...`
- `chore: ...`

## Что считается обязательным

- Для изменений, влияющих на сетевой протокол, добавьте заметки в `docs/PROTOCOL.md`.
- Для критических изменений в логике сервера добавьте тестовый сценарий или воспроизводимую проверку.
- Для изменений сборки, CI или релизов добавьте комментарий в `CHANGELOG.md`.
