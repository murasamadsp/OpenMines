# Сетевой протокол

> Это внутренний протокол Mines legacy-клиента и может меняться вместе с клиентской частью.

## Базовый wire-формат

Сетевая шина построена вокруг структуры `server/protocol/mod.rs::Packet`:

- `length: i32 LE` — общий размер кадра в байтах (включая этот префикс).
- `data_type: u8` — тип пакета (`U`, `B`, `J`, ...).
- `event_name: [u8; 2]` — двухсимвольный event.
- `payload: Vec<u8>` — необработанный полезный байтовый буфер.

Запись пакета:

```
[4 bytes length][1 byte data_type][2 bytes event][payload]
```

На чтении `Packet::try_decode` ждёт минимум 7 байт (`length` + `data_type` + `event_name`).

## Клиентские TY-пакеты

Для ряда игровых команд используется вложенный формат (`server/protocol/packets.rs`):

- `event_name: [u8;4]`
- `time: u32 LE`
- `x: u32 LE`
- `y: u32 LE`
- `sub_payload`

Этот формат декодируется через `TyPacket::decode`.

## Набор входящих/исходящих событий

Набор событий определяется в `server/protocol/packets.rs`.  
Ключевые категории:

- `AU`, `AH` — handshake/авторизация.
- `PI`, `@T`, `@L`, `P$`, `@B` — служебные и игровой статусы.
- `mO`, `mL`, `mU`, `mN` — чат.
- `HB` с саб-пакетами (`M`, `X`, `O`, `F`, `D`, `C`) — обновления мира и эффектов.
- `TY`-обёртка для действий пользователя (`XMOV`, `XBLD`, `XDIG` и т.д.).

Полный перечень и форматы декодеров лучше брать из `server/protocol/packets.rs`, чтобы не расходиться с реальным кодом.

## Обработка и безопасность

- Любые изменения формата пакета должны синхронизироваться между `server/protocol` и Unity-клиентом.
- Неверная длина/слишком большие пакеты считаются ошибкой протокола и закрывают сессию.
- На стороне клиента полезно считать, что `event_name` чувствителен к регистру (пример: `"cf"`).

## Диагностика

- Логи пакетов и ошибок протокола доступны в `tracing`-логах сервера.
- При тестировании клиента полезно смотреть `ConnectionManager` в `client/Assets/Scripts/ConnectionManager.cs`.
